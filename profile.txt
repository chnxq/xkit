整体结构

- `xkit` 是当前代码生成与脚手架中枢，负责根据 proto、ent schema、模板与约定生成后端代码，并逐步吸收已验证的通用生成规则。
- 主业务后端是 `admin`，主前端是 `admin-ui`。
- `xkit-template` 负责相对稳定的后端项目骨架。
- `x-crud`、`xkitpkg`、`xkitmod`、`x-utils`、`x-swagger` 是生成器与业务项目共同依赖的基础/工具仓库。

设计约束

- 生成代码与手写代码必须有清晰边界，避免下次生成时覆盖业务实现。
- 重复出现的“项目级手工修补”，应优先回归到 `xkit`、`xkit-template` 或共享库，而不是长期留在目标仓库里。
- 目标项目中的运行时装配、任务执行器、repo 扩展、默认数据等，要优先通过扩展点表达，而不是直接修改纯生成文件。

代码生成策略

- `*.gen.go`
  - 视为生成器拥有的文件。
  - 重新生成时允许被覆盖。
  - 原则上不应承载长期手写业务逻辑。
- `*_ext.go`
  - 视为扩展文件。
  - 通常由生成器“只在缺失时创建一次”。
  - 后续主要由人工维护，不应在普通生成中被覆盖。
- `internal/bootstrap/*_ext.go`
  - 也是稳定扩展点。
  - 用于补 generated wiring 之外的真实项目装配逻辑。
- 判断一个文件能否安全修改，先看文件头，再看是否属于 `.gen.go` / `_ext.go`。

文件头写入策略

- 来自 `xkit-template` 的文件，头部应标明：
  - `Code generated from: xkit-template.`
- 来自 `xkit` 动态生成的文件，头部应标明：
  - `Code generated by xkit.`
- 手写扩展文件不应伪装成纯生成文件。

模板划分准则

- `xkit-template`
  - 负责稳定、通用、跨项目可复用的后端骨架。
  - 例如：
    - `cmd/server`
    - `configs`
    - `internal/bootstrap`
    - `internal/server`
    - 一部分稳定的 `internal/data/bootstrap`
- `xkit`
  - 负责和 schema / proto / 项目配置强相关的动态生成内容。
  - 例如：
    - repo / service / server wiring 的动态部分
    - API 客户端与接口代码
    - 依据 schema 变化的 CRUD 代码
    - 基于配置生成的字段、过滤、排序、导出等行为
- 判断一个能力该放模板还是生成器：
  - 若它跨项目稳定存在、与具体 schema 关系弱，优先进 `xkit-template`。
  - 若它强依赖 schema / proto / 项目配置，优先进 `xkit`。
  - 若它只属于当前业务项目，应先落在目标项目扩展层，再评估是否具备抽象价值。

仓库定位

- `admin`
  - XAdmin 主后端。
  - 当前重点除了多租户稳定化，还包括 task runtime、默认数据初始化、审计与权限边界的收敛。
- `admin-ui`
  - 主前端，当前分支 `xadmin-api-integration`。
  - 已从基础 API 对接阶段进入任务管理、统一列表样式、cron 编辑体验和细节打磨阶段。
- `xkit`
  - 当前既承担项目生成，也承担已验证通用规则的回归。
  - 最近重点包括：
    - task 结构与配置对齐
    - 日志时间过滤生成规则
    - “仅生成时间变化不应重写文件”的优化
- `xkit-template`
  - 用于沉淀后端基础骨架。
