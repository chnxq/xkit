Project Profile

- `xkit` is the current code generation hub for XAdmin.
- Primary backend project: `admin`
- Primary frontend project: `admin-ui`
- Stable backend bootstrap/template baseline: `xkit-template`
- Shared dependency repos commonly involved in generation and runtime:
  - `x-crud`
  - `xkitpkg`
  - `xkitmod`
  - `x-utils`
  - `x-swagger`

Current Positioning

- `xkit-template` owns relatively stable backend project skeleton content.
- `xkit` owns dynamic generation that depends on schema, proto, and project config.
- `admin` remains the real verification target for backend generation rules.
- `admin-ui` remains the real verification target for frontend meta generation rules.
- For current frontend generation work, `xkit/examples/admin/admin-target-config/admin.yaml`
  is the operational source of truth.

Generation File Boundary

- `*.gen.go`
  - generator-owned files
  - can be overwritten during regeneration
  - should not carry long-term handwritten business logic
- `*_ext.go`
  - extension files
  - typically created once if missing
  - expected to hold handwritten project-specific logic
- `internal/bootstrap/*_ext.go`
  - stable extension points for runtime/bootstrap assembly
- Generated frontend meta files under `admin-ui/apps/web-antd/src/views/generated/admin/`
  are generator-owned outputs and should be treated like other generated artifacts.

Template vs Generator Split

- Put code in `xkit-template` when it is:
  - stable across projects
  - weakly coupled to schema/proto
  - part of backend project skeleton/bootstrap
- Put code in `xkit` when it is:
  - strongly driven by schema/proto/config
  - resource-specific
  - expected to change when CRUD/resource metadata changes
- Keep code in target-project extension files first when it is:
  - business-specific
  - not yet proven reusable
  - still being validated in a real project

Current Frontend Config Conventions

- Real iteration baseline:
  - `xkit/examples/admin/admin-target-config/admin.yaml`
- Synced reference configs:
  - `xkit/examples/admin/admin-config/admin.yaml`
  - `xkit/examples/admin-v2/admin-config/admin.yaml`
- `frontend.list.columns`
  - may use only the field name
  - may also provide explicit titles
  - if titles are omitted, generation should fall back to schema-derived names/comments
- `frontend.form.fields`
  - follows the same title fallback idea as list columns
- `frontend.list.filters`
  - should stay form-like rather than list-column-like
  - if only field/component is provided, reuse field title fallback
  - if explicit titles are provided, use those titles for i18n

Current Placeholder / I18n Rules

- zh-CN generated placeholders should not contain mixed English artifacts such as:
  - `Search ...`
  - `Select ...`
  - `... Range`
- For zh-CN generation, placeholder text should fall back to the field label.
- Enum i18n should not be forced out of xkit inference when the source is better maintained
  as external language data; copied language resources under generated `langs/` are the
  current practical direction.

Operational Guidance

- Start config work from `admin-target-config`, then sync back to reference configs only after validation.
- Avoid broad manual rewrites of large YAML files; prefer narrow edits plus immediate regeneration.
- When a target project repeatedly needs the same handwritten patch, first evaluate whether it belongs in:
  - shared template code
  - generator logic
  - generated shared frontend resources
- For frontend rollout, prefer this order:
  - generate meta
  - wire page `index.vue`
  - preserve only business-specific slots/overrides in handwritten page code
