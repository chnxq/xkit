## `xkit` Design Intent

`xkit` is the code generation toolchain and should be treated as the place where repeatable generation rules live. Its job is not just to emit files once, but to make regeneration predictable after data model, proto, template, or project configuration changes.

Directory map:

```text
xkit/
|-- cmd/
|   `-- xkit/                 # CLI entry point
|-- doc/                      # generation/integration notes
|-- examples/
|   |-- xadmin/               # backend example/config used during this work
|   `-- xadmin-web/           # frontend/example material, not used for Vben generation
`-- internal/
    |-- binding/              # binding/model helpers
    |-- cli/                  # CLI orchestration
    |-- codegen/
    |   `-- template/         # template rendering and file output logic
    |-- config/               # generator config model/loading
    |-- entschema/            # ent/schema parsing and interpretation
    |-- project/              # project layout and generation planning
    |-- proto/                # proto generation/planning support
    |-- scaffold/             # project scaffolding
    `-- sourceimport/         # source/template import helpers
```

Core design principles:

- Configuration and model definitions should be the source of truth wherever possible.
- Repeated backend patterns should be fixed in the generator, template pipeline, or shared libraries instead of being patched repeatedly in generated projects.
- Generator output should have clear provenance markers so code from `protoc`, `xkit`, `xkit-template`, and Codex/manual extension files are not confused.
- Generated code should support safe extension points, normally via `_ext.go` files or explicit special methods, so regeneration does not overwrite business-specific logic.
- Cross-project correctness issues, such as mapper enum conversion, audit field population, and CRUD behavior for standard fields, should be solved in `xkit`, `x-crud`, `x-utils`, `xkitmod`, or templates according to ownership.
- Generation should be deterministic enough that regenerated `admin` can be compared against previous `xadmin`/backup output and missing functions can be found systematically.

Practical implication:

- If a defect is caused by generated CRUD, mapper behavior, template bootstrap, or common middleware, prefer fixing the generator/template/shared package first.
- Local backend edits in `admin` are acceptable for iteration, but should be candidates for promotion into `xkit` or `xkit-template` once stable.

## `xkit-template` Design Intent

`xkit-template` is the backend starter/runtime template. It should contain reusable project skeleton code that every generated service should inherit.

Directory map:

```text
xkit-template/
|-- cmd/
|   `-- server/               # template server entrypoint/assets wiring
|-- configs/                  # starter configuration files
`-- internal/
    |-- bootstrap/            # generated service/data construction glue
    |-- data/
    |   `-- bootstrap/        # data initialization/bootstrap helpers
    |-- server/               # HTTP/gRPC server wiring, middleware, manual hooks
    `-- service/              # service layer skeleton and extension points
```

Core design principles:

- Provide stable application wiring: config loading, server bootstrap, logging, data/repository construction, middleware, audit plumbing, migration/default data hooks, and generated service registration.
- Keep template code generic and reusable. Project-specific menu definitions, API lists, or admin-specific behavior should either be parameterized by `xkit` or placed in clearly marked extension files.
- Avoid hiding business logic in generated boilerplate. Template code should expose explicit extension points for app-specific service behavior.
- Preserve regeneration safety: files copied from the template should be clearly marked as coming from `xkit-template`, and direct Codex/manual additions should be marked separately.
- The template should encode decisions that are stable across generated projects, such as how CRUD fills `created_at`, `created_by`, `updated_at`, `updated_by`, and how optional logging/audit features are wired.

Candidates already identified for template promotion:

- API audit and login audit logging wiring controlled by config.
- Default/demo SQL data generation and migration support.
- Manual/admin portal service patterns that are broadly useful.
- Safe extension registration for routes not produced directly by proto generation.
- Common sync patterns for backend-managed resources.
