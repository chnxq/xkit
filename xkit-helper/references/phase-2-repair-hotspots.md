# Phase 2 Repair Hotspots

Use this checklist after Phase 1 if the new project still differs from the chosen reference implementation.

## 0. Resolve the reference project first

The repair phase must start from a local reference directory.

Allowed inputs:

- local path, for example `D:\GoProjects\XAdmin\admin`
- git repository URL

If a git URL is provided:

- clone it to a local working path first
- use the cloned directory as the reference repo root
- record the exact path and revision used during the repair session

For the rest of this document, call that directory `<ReferenceProjectRoot>`.

## 1. Preserved server files fell back to template stubs

### Highest-risk files

- `internal/server/manual_http_data.go`
- `internal/server/options.go`
- `internal/server/http_options.go`
- `internal/server/grpc_options.go`
- `internal/server/sse.go`

### Typical symptoms

- login, captcha, refresh token, profile, analytics, navigation, or SSE behavior is missing
- `internal/server` tests fail because manual types or helpers are gone
- message service cannot resolve shared SSE server

### Repair

Diff against `<ReferenceProjectRoot>` and bring over the current hand-written implementation, then rewrite imports from the reference module path to `<Module>/...`.

## 2. Extension helper names now collide with generated code

### Typical symptoms

- compile errors like `redeclared in this block`

### Confirmed examples from qadmin

- `permissionFieldMaskContains`
- `roleFieldMaskContains`

### Repair

Check whether generated code now defines the same helper. If it does, remove or rename the extension-side helper instead of patching unrelated generated files first.

## 3. Ent query types are missing `Modify(...)`

### Typical symptoms

- repo package build errors mentioning `missing method Modify`
- failures inside `BuildListSelectorWithPaging`

### Main repair file

- `internal/data/ent/query_modify_ext.go`

### Types that needed coverage in qadmin

- `ApiAuditLogQuery`
- `DataAccessAuditLogQuery`
- `DictEntryQuery`
- `DictTypeQuery`
- `FileQuery`
- `InternalMessageCategoryQuery`

Scan all generated repo files and cover every list-builder query type in one pass.

## 4. Hand-written business features commonly required from the reference project

Compare whether the target project also needs these groups from `<ReferenceProjectRoot>`:

- auth password verification and token handling
- captcha support
- viewer auth middleware
- manual HTTP services for login, admin portal, profile, analytics, and menu sync
- internal message repo/service extensions
- audit log repo wrappers and DB logging integration
- default data bootstrap extensions
- SQL default/demo data

## 5. Generated files may still contain reference business behavior

Do not assume every `*.gen.go` difference is safe to ignore.

The reference project may have been generated by an older generator or manually adjusted before the boundary between generated code and extension code was stabilized. If a regenerated target compiles but behaves differently, inspect the relevant generated files before deciding that Phase 2 is done.

Confirmed high-risk example:

- `internal/data/repo/user_repo.gen.go` in the reference project may load `UserRole`, `UserOrgUnit`, or `UserPosition` relations and fill DTO fields such as `role_ids`
- a freshly generated target may only return the base `User` row from `Get`, leaving `role_ids` empty
- portal navigation and permission loading that call `userRepo.Get(...).GetRoleIds()` will then return no menus even after login
- this can make a frontend backend-access route such as `/analytics` appear missing even though `src/views/dashboard/analytics/index.vue` exists and the Vite history fallback returns HTTP 200

Repair choices:

- move the reference behavior into an extension-owned reader such as `ListRoleIDsByUserID(ctx, userID)` and make portal/viewer auth use that relation reader
- or bring over the reference generated relation-loading behavior only as an intentional compatibility patch, with a note that it is a generator/template debt
- add tests that prove role IDs from `sys_user_roles` drive permission/menu lookup instead of relying on an empty DTO field

## 6. Backend menu and frontend route contract

When the frontend uses backend access mode, route existence is decided by the backend menu response, not only by files under `src/views`.

Verify the contract explicitly:

- frontend preference sets backend access mode, for example `accessMode: 'backend'`
- frontend `homePath` such as `/analytics` is present in `/admin/v1/routes`
- backend `defaultNavigationRoutes()` or seeded menus contain the same path and redirect target
- `GetNavigation` has a deliberate empty-result behavior; if ordinary users can temporarily have no menu IDs, return a safe default route or fail with a clear authorization state
- each backend menu component path maps to an actual frontend page, for example `/dashboard/analytics/index` -> `apps/web-antd/src/views/dashboard/analytics/index.vue`
- skip framework/layout components such as `BasicLayout` and `IFrameView` during the component-file scan

Suggested PowerShell component scan:

```powershell
$frontendRoot = 'D:\GoProjects\XAdmin\admin-ui\apps\web-antd\src\views'
$backendFiles = @(
  '<target>\internal\server\manual_http_data.go',
  '<target>\internal\data\repo\menu_repo_ext.go'
)
$components = foreach ($file in $backendFiles) {
  Select-String -Path $file -Pattern 'Component:\s*(?:ptr\()?"([^"]+)"' |
    ForEach-Object { $_.Matches[0].Groups[1].Value }
}
$components | Sort-Object -Unique | Where-Object {
  $_ -notin @('BasicLayout', 'IFrameView') -and $_.StartsWith('/')
} | ForEach-Object {
  $expected = Join-Path $frontendRoot ($_.TrimStart('/').Replace('/','\') + '.vue')
  if (-not (Test-Path -LiteralPath $expected)) { "$_ -> missing $expected" }
}
```

## 7. Verification target

At minimum:

```powershell
go test ./...
```

The practical success condition is:

- target repo compiles
- bootstrap and core runtime paths are present
- target repo is behaviorally close enough to the resolved reference project for the intended iteration
- the real process is written into `<target>/readme.md`
